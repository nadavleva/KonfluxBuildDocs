## Issue 1: Missing Red Hat Registry Credentials (Team-Managed Solution) 

**Status:** ✅ RESOLVED  
**Priority:** CRITICAL  
**Owner:** Solution implemented  
**Date Created:** 2026-05-19  
**Updated:** 2026-05-20 - Solution: Service account token + Kubernetes secrets

### Solution Implemented

✅ **RESOLVED** - Buildah credential passing solved using service account token approach

#### Method

1. **Generate Service Account Token** (for brew.registry.redhat.io):
   - Follow instructions: https://source.redhat.com/groups/public/teamnado/wiki/brew_registry
   - Obtain username and password/token from Red Hat internal wiki

2. **Create Two Konflux Secrets**:
   - `brew-secret`: Contains credentials for `brew.registry.redhat.io` (internal Red Hat build registry)
   - `image-pull-redhat-io`: Contains credentials for `registry.redhat.io` (public Red Hat registry)

3. **Link Secrets to Builder Service Account**:
   - Configure imagePullSecrets in builder service account
   - Secrets are automatically mounted and available in build pods
   - Buildah can access them via standard Docker auth mechanisms

#### Why This Works

- Service account tokens are **not personal credentials** - they're team-managed and persistent
- Both Kubernetes pod pulls and buildah container builds can access the secrets
- Secrets are automatically injected into the build environment
- Follows Red Hat's standard approach for enterprise builds

#### Reference Resources

- **Internal Wiki**: https://source.redhat.com/groups/public/teamnado/wiki/brew_registry
- **Konflux Documentation**: Service account and secret management
- **Related Secrets**: Created in `rh-ocp-dr-tenant` namespace:
  - `brew-secret` - For `brew.registry.redhat.io` access
  - `image-pull-redhat-io` - For `registry.redhat.io` access

#### Related Follow-Up Issue 2

- **Issue 2**: Team-managed credentials generation (service account vs. personal tokens)
- **Issue 3**: Base image optimization (ubi9/ubi-micro for reduced footprint)

## Issue 2: Team-Managed Credentials Generation (Non-Developer Tokens)

**Status:** 🟡 OPEN - Implementation Pending  
**Priority:** HIGH  
**Owner:** TBD  
**Date Created:** 2026-05-20  
**Blocks:** Production credential rotation strategy
**Story:** https://redhat.atlassian.net/browse/VIRTDR-143

### Problem

Current solution uses personal service account tokens bound to individual developers:
- Personal tokens are developer-specific
- Token lifecycle tied to developer accounts
- Difficult to rotate without developer involvement
- Not suitable for CI/CD automation in production

### Requirement

Need a mechanism to generate **team-managed, persistent credentials** for Konflux builds that:
- Are **not tied to individual developers**
- Can be automatically rotated by the team
- Support dedicated service accounts for build infrastructure
- Are suitable for unattended CI/CD environments

### Recommended Solution: Team Bot Account

Per Konflux documentation and best practices:

1. **Create a Team Bot Account:**
   - Create a new Red Hat corporate account using a shared team email address or Google Group
   - Example: `rh-ocp-dr-team@redhat.com` or use existing team distribution list
   - This bot account is **decoupled from individual developer accounts**

2. **Generate Credentials via Bot Account:**
   - Log into Red Hat Customer Portal using the bot account: https://access.redhat.com/terms-based-registry/
   - Log into Brew Registry portal using the bot account: https://source.redhat.com/groups/public/teamnado/wiki/brew_registry
   - Generate registry service account tokens **from the bot account**
   - These tokens are now owned at the team level, not individual level

3. **Update Konflux Secrets:**
   - Use the bot-generated tokens to create/update Image Pull Secrets in Konflux
   - Replace existing personal credential secrets with bot-generated ones
   - Update `brew-secret` and `image-pull-redhat-io` with bot tokens

4. **Team-Level Quay Credentials (if needed):**
   - For Quay robot accounts (e.g., redhat-services-prod), submit Merge Request to internal `quay-access-management` GitOps repository
   - Quay handles team credentials through a separate process

### Important Notes

- **"Registry Service Account" tokens are correct:** The Konflux documentation's advice against "personal credentials" means don't use raw Red Hat SSO username/password (which any namespace admin can read in plaintext)
- **Bot-generated tokens are the solution:** Using registry service account tokens generated by a bot account is exactly the recommended approach
- **Token lifecycle:** Bot-generated tokens can be rotated by the team without individual developer involvement
- **Team ownership:** Secrets created from bot-generated credentials remain valid and manageable even if individual team members change

### Implementation Tasks

- [ ] Create shared team email/Google Group for bot account (e.g., rh-ocp-dr-team@redhat.com)
- [ ] Log into Red Hat portal with bot account and generate new service account tokens
- [ ] Log into Brew registry portal with bot account and generate new brew tokens
- [ ] Update existing `brew-secret` Konflux secret with bot-generated credentials
- [ ] Update existing `image-pull-redhat-io` Konflux secret with bot-generated credentials
- [ ] Document bot account procedures for team
- [ ] Set up team rotation schedule for credential renewal (recommended annually)

### Success Criteria

- ✅ Team-managed service account for `rh-ocp-dr-tenant` builds (bot-generated)
- ✅ Credentials completely decoupled from personal developer accounts
- ✅ Token can be rotated without individual developer involvement

## Issue 3: Base Image Optimization (UBI9/UBI-Micro)

**Status:** 🟡 OPEN - Optimization  
**Priority:** MEDIUM  
**Owner:** TBD  
**Date Created:** 2026-05-20  
**Dependency:** Issue 1 (credentials solution)
**Story:** https://redhat.atlassian.net/browse/VIRTDR-142

### Problem

Currently using `registry.redhat.io/ubi9/ubi-minimal` as runtime base image:
- ~150MB uncompressed size
- Includes unnecessary packages for minimal operator image
- Larger container pulls and registry storage

### Opportunity

Red Hat now offers `ubi9/ubi-micro` - ultra-minimal container image:
- Significantly smaller footprint (~50MB)
- Optimized for containerized workloads
- Same security baseline as ubi-minimal
- Already widely adopted in Red Hat products

### Proposed Solution

Replace runtime base image in [hub.Dockerfile](hub.Dockerfile#L47):

**Current:**
```dockerfile
FROM registry.redhat.io/ubi9/ubi-minimal@sha256:12db9874bd753eb98b1ab3d840e75de5d6842ac0604fbd68c012adefe97140be
```

**Proposed:**
```dockerfile
FROM registry.access.redhat.com/ubi9/ubi-micro:9.8-1777876409
# or digest: sha256:13e4883c409d19ddd8816dcda0a0ee1830084e1d84e501f05f7b60ecc05d48c2
```

### Changes Required

1. Update [hub.Dockerfile](hub.Dockerfile#L47) to use ubi-micro
2. Test operator functionality with minimal runtime image
3. Update cluster operator Dockerfile similarly
4. Benchmark image pull time and registry storage savings

### ✅ Decision: RHDR Naming Convention Finalized

✅ **Naming Convention**: `rhdr` (Red Hat Disaster Recovery)
- **Status**: ✅ **VALIDATED & APPROVED** - Confirmed as available and not in use (2026-05-25)
- **Applies to**: All operator names, repositories, container images, and infrastructure  
- **Examples**: `rhdr-hub-operator`, `rhdr-multicluster-operator`, `rh-ocp-dr/rhdr-*`
- **Key Advantage**: Complete independence from ODF's `odr` downstream naming scheme  

### Problem (Now Resolved)
To completely isolate the Virtual DR (VDR) project from the standard ODF payload stack, we moved from shared repository contexts to a structured, independent multi-repository model with a unique naming boundary.

**Original Options Considered**:
- `odr` - Aligned with ODF but implied ODF dependency ❌
- `rdr` - Clean but less clear to external teams ❌
- `ramen` - Product name but verbose in configs ❌
- `rhdr` - Clear brand identity, distinct from ODF, completely independent ✅ SELECTED
